# 제안서 — 집계 상태 요약 띠 (Status Summary Strip)

> 근거: 메인페이지 UX 감사(2026-06-30). 증거: `artifacts/responsive-screenshots/main-page-audit/2026-06-30_main/` (03 데스크톱·05 모바일)

## 1. 현황
- 지도가 로드되면 공항이 **flight-category 색 마커 + wind barb**로 표시됨(05에서 RKNY 주황 등) — *개별* 상태는 보임.
- 그러나 **집계·한눈 요약이 없음**: "지금 IFR 공항 몇 개? 경보 몇 건? 다음 변화는 언제?"를 알려면 마커를 일일이 스캔해야 함(감사 finding #7).

## 2. 문제 (헌법 §6 정보 우선순위 위배)
헌법이 정한 화면 우선순위 중 ②현재 운영 상태(집계)·③다음 중요한 변화가 첫 화면에 **직접 노출되지 않음**. 시간 압박 속에서 "전체가 안전한가?"를 즉답하지 못함.

## 3. 제안

### 내용 (무엇을 보여주나)
1. **카테고리 분포** — VFR/MVFR/IFR/LIFR 공항 카운트(헌법 표준색 + 숫자, 색만으로 구분 금지 → 숫자·라벨 병기).
2. **경보 공항 수** — 활성 공항경보/SIGMET·AIRMET 영향 공항 수.
3. **다음 임박 변화** — 가장 가까운 TAF 카테고리 전환(공항 + 시각).

### 위치/형태 (옵션)
| 옵션 | 설명 | 평가 |
|------|------|------|
| **A. 지도 상단 오버레이 띠** (추천) | 지도 위 좁은 띠, 데이터 레이어와 분리 | 진입 즉시 보임, 지도 가림 최소 |
| B. 사이드바 확장 시 상단 요약 | 확장해야 보임 | 기본(축소) 상태에선 안 보임 ✗ |
| C. 토글식 패널 | 클릭해야 열림 | "상태 먼저" 원칙과 충돌 ✗ |

→ **옵션 A 권장.** 차분한 중립 배경 + 카테고리 색 점만 사용(헌법 §2 "데이터가 주인공", 베이스맵 위 떠 보이게).

### 데이터
- 이미 `weatherData`(airports + metar + category, backend `flight-category.js`)에 전부 존재. **새 API 불필요.** 집계는 프론트 파생 모델로 계산.

### 반응형
- 모바일은 이미 `MobileMapOverlay`(상단 레이어 버튼)가 있음 → 그 위/아래에 축약 요약(예: "IFR 2 · 경보 1") 1줄로 통합. 데스크톱 띠를 그대로 축소하지 않음(헌법 §6 모바일 원칙).

## 4. 결정 필요 (열린 질문)
1. **위치** — 옵션 A(지도 상단 띠) 확정?
2. **클릭 인터랙션 범위** — 카테고리 클릭 시 해당 공항 지도 하이라이트까지? 아니면 표시-전용(MVP)?
3. **"다음 변화" 포함 시점** — TAF 전환 계산은 브리핑 로직과 일부 겹침. MVP는 카테고리 분포 + 경보 수만, "다음 변화"는 2단계?

## 5. 기대 효과
- 첫 화면에서 **전체 안전 상태 즉시 인지** → 도구의 핵심 가치를 진입 즉시 전달(헌법 §2-5 "상태 먼저").
- 마커 시각 스캔 의존 제거 → 판독 시간 단축, 오판 위험 감소.
- 기존 데이터 재사용으로 백엔드 변경 0.

## 6. 다음 단계
승인 시: (1) 옵션 A 목업 1장 + 파생 모델 스펙 → (2) 캡처 기준선 → (3) 구현 → (4) 스크린샷 회귀 + AA 대비 검증.
