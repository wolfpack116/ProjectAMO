## A. 신뢰·메타 묶음

세 페르소나(IFR·디스패처·예보관)가 공통으로 "**지금 보는 이 값이 언제 것인지, 어디서 왔는지, 원문이 뭔지**를 못 봐서 최종 결정에 못 쓴다"고 했다. 신규 데이터가 거의 필요 없는데 신뢰 회복 효과가 가장 크다 → **최우선.**

### 1. 출처·발표시각·유효시각 무조건 명시
- **무엇**: 모든 관측/예보/NOTAM에 **출처(예: KMA apihub / KOCA) · 발표시각 · 유효시각(있으면)**을 항상 그대로 표기. 값을 "오래됐다"고 판정하는 신호등·경보는 두지 않는다 — 시각을 정직하게 보여주면 오래됐는지는 **사용자가 판단**한다.
  - (신호등 신선도 점수는 임계값을 우리가 정의·유지해야 하고 "무엇이 stale인가" 논쟁을 부른다 → 범위에서 제외.)
- **누구·왜**: IFR·디스패처·예보관 공통. "이 값이 언제 것인지·어디서 왔는지"만 보이면 스스로 신뢰도를 매길 수 있다. 판정은 사용자 몫.
- **데이터·범위**: 발표/수신/유효 시각과 출처는 이미 store·파이프라인에 있음 → **표시만.** 공용 메타 컴포넌트 하나로(#4와 공유). 타입별 임계값 정의 불필요.
- **부수**: NOTAM 등 **수집 실패 시엔 마지막 수신시각을 그대로 노출**(최신인 척하지 않음). 필요하면 "마지막 수신: 12:30Z" 사실 표기만 — 심각도 판정 아님.
- **열린 질문**: 유효시각이 없는 데이터(관측 등)는 발표/수신시각만 표기로 충분한가? · 시각 표기 형식(Z/KST 병기?).

### 2. 원문 토글
- **무엇**: 재구성한 METAR/TAF/SIGMET 옆 "원문 보기" 토글(원본 TAC/IWXXM 라인).
- **누구·왜**: 조종사·예보관은 공식 앱 원문과 대조 가능 → 재구성본만 있으면 "파싱 누락 아닌가" 의심(IFR·예보관 지적). 원문을 스스로 검증하게.
- **데이터·범위**: **먼저 확인 필요** — 현재 파이프라인이 IWXXM만 받고 TAC 원문을 버리는지. 버린다면 원문 보존이 선행. 재조립본이면 "재구성" 라벨을 정직하게.
- **열린 질문**: 원문이 아예 없는 소스(IWXXM only)는 "원문 없음·재구성" 명시로 갈지?

### 3. 데이터 없음 명시
- **무엇**: 미지원 항목을 빈칸으로 두지 않고 "이 공항 AMOS 미수집" 같은 명시 칩.
- **누구·왜**: AMOS/이륙예보/공항정보가 8개 공항(RKSI/RKSS/RKPC/RKPU/RKJY/RKJB/RKNY/RKPK)만인데, 지방공항 운영자는 **빠진 걸 모른다**(침묵 누락 — IFR·VFR·회전익 3명 공통, 가장 위험).
- **데이터·범위**: 지원 공항 목록은 config에 있음. 뷰에서 "지원/미지원"을 구분해 렌더.
- **열린 질문**: "미수집"과 "일시적 로드 실패"를 구분 표기할지?

### 4. 모든 메타(모델 신원·발표시각·해상도)
- **무엇**: NWP/차트에 모델명(GDPS/RDPS 등)·발표시각·초기장 cutoff·해상도를 명시. 모든 표·차트 상단에 발표/유효/만료 메타 고정.
- **누구·왜**: 예보관은 "이 바람장이 어느 모델·해상도인지" 못 보면 신뢰도를 줄 수 없음. 디스패처는 자동계산 시스템과 값이 다를 때 어느 출처가 정답인지 판단 불가.
- **데이터·범위**: KIM 수집 시 manifest에 run/해상도 정보 존재 가능성 높음 → 뷰 노출. #1과 공용 메타 컴포넌트.
- **열린 질문**: KIM manifest에 모델 계열/해상도가 실제로 들어있나(확인 필요) · 없으면 수집 단계 보강 필요.

### 6. TAF 표시방법 일원화
- **무엇**: 공항패널 TAF 탭과 브리핑 목적지 TAF의 표시를 **하나의 공유 뷰모델/컴포넌트**로 통일(그리드/타임라인/기간표 + 원문 라인 + 카테고리 색띠).
- **누구·왜**: 지금 두 곳의 TAF 표현이 갈라져 유지보수·학습 비용↑. 예보관은 "변화군 원문 + 시간축 색띠 나란히"를 원함. 전원이 TAF를 본다.

### 7. 로그인 + 개인 미니마 (+ 역할 · 예보관 문의)
> 📎 **구현 참조 별도 문서**: [login-accounts-reference.md](2026-07-05-login-accounts-reference.md) — 아키텍처·세션·비번해시·DB 스키마·API 명세·인가·검증·**예보관 문의 큐**·보안 체크리스트·구현 단계를 세세하게(스펙 리뷰 반영). 스펙/구현 시 그 문서를 근거로.
- **무엇**: 고전적 로그인(아이디+비밀번호). 사용자별 프리셋(개인 미니마·저장 경로)을 서버에 저장. 역할(조종사/예보관, 예보관은 **승인/초대제**). **조종사가 [예보관에게 문의]하면 예보관 콘솔에 팝업/대기열로 뜸** → 예보관이 그 경로·브리핑을 열어 응대.
- **누구·왜**: 미니마·프리셋을 자기 기준으로 저장·재사용(참조문서 §3.1 알림 트리거의 기준선). 예보관은 조종사가 고른 경로를 **바로 화면에 받아** 응대(구두 전달·코드 불필요). #13 알림의 토대.
- **★ 1차 목표 (발표용, 지금)**:
  - 고전 로그인(**소셜 X**) + 프리셋/경로 **서버 저장**
  - 역할 2종(조종사/예보관, 예보관 승인제)
  - **예보관 문의 큐**: 조종사 [문의] → 예보관 콘솔 팝업(폴링) → 경로·브리핑 열람 (**외부 수신 데이터만**)
- **차후 (발표 후)**:
  - **내부망→외부 일방향 데이터 내보내기 + 역할 기반 2계층** → 예보관이 더 풍부한 자료 열람 (회사 협의·거버넌스 필요, egress만·인바운드 없음)
  - 소셜 로그인, 다중기기 동기화, 예보관 공항 담당 배정, 실시간 SSE

- **열린 질문**: 예보관 등록(관리자 생성 vs 초대토큰) · 마이그레이션 충돌 기본값 · SQLite→Postgres 전환 시점.

### 8. 공항 접근 최저치 + 현재 카테고리 자동대조
- **이미 있는 부분**: 사용자 설정 공항별 미니마(시정m·운고ft) 입력·저장이 [SettingsModal.jsx](../../../frontend/src/features/settings/SettingsModal.jsx) "공항 미니마" 탭 + localStorage `airport_minima_settings`로 존재. → "내 개인 기준" 대조는 이걸로.
- **새로 넣을 부분(1단계, 라이트)**: 그와 별개로 **AIP 절차별 published minima**(ILS/RNAV DA·MDA·RVR)를 앱에 담아, 현재/예보 카테고리와 자동 대조해 "GO/한계/불가" 판정. IFR **채택 블로커 1순위**였던 항목.
- **데이터·범위 (부담 작음)**:
  - **자동 스크래퍼 만들지 않는다.** 한국 주요 공항(현 8개+α)만 대상 → 접근절차 다 합쳐 수십 개.
  - KOCA(aim.koca.go.kr) AIP 접근 차트를 보고 **최저치를 손으로 구조화 파일(JSON)에 전사** → 앱은 그 파일과 현재 날씨를 대조만.
  - **AIRAC 28일 주기 갱신**은 사람이 그 파일만 확인해 수정(공항 몇 개라 몇 분). 자동 추출·상시 파이프라인 없음.
- **왜 라이트가 맞나**: PDF 차트에서 최저치 자동 추출은 만들고 검증하는 게 더 비싸고, DA 숫자 오류는 안전 리스크. 대상이 작으니 수작업 전사가 더 빠르고 정확.
- **열린 질문**: ⚠️ **한국 AIP 데이터를 이 도구에 담아 쓰는 게 사용범위/라이선스상 가능한지** KOCA/기상청 확인(공개 자료라도 재사용 조건 있을 수 있음). · 카테고리(A~D)·조건부(등화고장 등) 최저치를 어디까지 담을지(1단계는 대표값만?).

### 13. 경로 브리핑 예보변화 알림 (푸시)
> 📎 **구현 참조 별도 문서**: [route-forecast-alert-reference.md](2026-07-04-route-forecast-alert-reference.md) — 선례(ACAS 등) 파라미터, ICAO 임계값 표, diff 엔진 의사코드, 데이터 모델, 알림피로 플레이북을 "그대로 이식" 수준으로 정리. 스펙 작성 시 그 문서를 근거로. **(2026-07-07 개정됨 — 현재상태 반영)**
> **📌 2026-07 현재상태:** 아래 "하드 전제"(#7 로그인·서버 계획저장·서버 미니마)는 **이미 배포됨**(routes·presets 테이블, auth). 감시 소스는 국내(KMA)+해외(NOAA) 2축으로 갈림. **남은 신규 = 재브리핑 스케줄러·diff 엔진·Web Push·알림 테이블.** 상세는 참조문서 §5B·§6·§7·§8·§9.
- **무엇**: 사용자가 **비행예정일 + 경로**를 미리 저장해두면, 그 날이 오기 전까지 **해당 경로·시각의 예보가 의미있게 바뀔 때** 푸시로 알린다. (모니터링 페이지의 "공항 실시간 임계값 경보"와는 다른 기능 — 그건 공항 단위·현재값 기준, 이건 **내 저장된 비행 단위·예보변화 기준.**)
  - 예: 목적지 TAF가 내 ETA 시각 기준 VFR→IFR로 하락 / 교체공항이 새로 "필요"로 뒤집힘(1-2-3) / 경로상 새 SIGMET·AIRMET 발효 / 엔루트 착빙·난류가 moderate+ 로 상승 / 측풍이 내 미니마 초과.
- **누구·왜**: 조종사·디스패처. 브리핑은 "지금 한 번 보는 것"인데, 비행은 며칠 뒤 → 그 사이 예보가 나빠지는 걸 **자동으로 감시**해주는 게 핵심 가치. 주 사용이 앱이므로 **앱이 꺼져 있어도 오는 푸시**가 목표.
- **데이터·범위 (핵심 설계)**:
  1. **계획을 서버에 저장** — 지금 저장 경로는 localStorage(`routeStore.js`, 클라이언트 전용). 앱이 꺼져 있어도 감시하려면 **서버가 계획(경로+ETD/ETA+미니마)을 알아야** 함 → 로그인·서버 저장(#7)이 **하드 전제**.
  2. **주기적 재브리핑 + 스냅샷 비교** — 상류 데이터 갱신 시(또는 스케줄) 각 저장 계획에 대해 `POST /api/route-briefing`을 재계산하고, **직전 스냅샷과 diff**. 상류 변화감지는 `store.js`(SHA-256)·snapshot-meta가 이미 함 → 트리거로 재사용.
  3. **"의미있는 변화" 규칙(diff 엔진)** — 새로 만들 부분. 단, 판정 로직은 **이미 있는 걸 재사용**: `flight-category.js`(카테고리 하락), `taf-window.js`(교체 필요 1-2-3 뒤집힘), `hazard-section.js`(경로∩시간∩고도 새 위험), enroute 모델(착빙·난류 상승). v1은 **카테고리 하락 + 교체필요 플립 + 경로상 새 SIGMET** 3종만, 이후 확장.
  4. **전달 = Web Push** — service worker + VAPID + 구독 저장 + 변화 시 발송. (앞선 조사에서 Phase 2로 미뤘던 부분이 여기선 **본체**.)
  5. **중복 억제·만료** — 같은 변화 재알림 금지(계획×변화종류 키로 쿨다운), 비행시각 지나면 감시 종료. 쿨다운·조용시간 개념은 모니터링 `alert-state.js`에서 차용.
- **참고 선례(실제 존재)**: **Leidos ACAS**(1800wxbrief) — 비행 저장 시 경로 새 악조건 알림, **거의 동일한 서비스**(고도필터 계획고도+4000ft, 감시시작 사용자설정). **ForeFlight Flight Alerts**(ETD 2h 전 감시, 유의미 변화 시 푸시). **Garmin**(비행계획+ETE+3h, 푸시 대신 인앱 하이라이트로 노이즈 회피). **SkyDemon**(경로 회랑 48h 실시간). **Weather Company Maverick Dispatch**(알림 피로를 통합·커스텀 임계로 해결).
- **선례가 준 설계 교훈(반드시 반영)**:
  1. **알림 피로 방지가 성패** — 재발행마다 알리지 말 것. **내용 변화 vs 행정적 재발행(내용 동일) 구분**(NWS Categorical Amendment Criteria 참고). 항목 ID+개정번호로 **중복 제거**, 조건종류별 **rate limit**(예: 종류당 5분 1회).
  2. **조건종류별 차등**: 신규 SIGMET/TFR/활주로폐쇄 = 즉시 푸시 / AIRMET은 **착빙·산악불명료만** / 일상 METAR·TAF 재발행(내용 동일) = 인앱만.
  3. **범위 = 경로 회랑 + 버퍼(±10~20nm) + 고도필터**(ACAS식 계획고도 위 초과분 제외).
  4. **감시 라이프사이클**: 시작 = ETD−3~6h, 종료 = ETA+1h 또는 사용자 취소(지난 비행은 감시 중단).
  5. **개인 미니마 대비 필터**(#7·#8 연계) — VFR/IFR/상업 허용치가 달라 사용자 미니마로 걸러야 함.
- **v1 최소 범위(선례 기준)**: 출발 전(ETD−수시간) **신규 SIGMET·TFR·활주로폐쇄 NOTAM + 목적지 카테고리 하락** 3종만. 인플라이트 감시·리라우팅 제안은 디스패치 영역이라 제외.
- **열린 질문**: TAF "유의미한 변화"의 정확한 임계(풍속 shift >10kt? 운고 <500ft? 시정 <3sm?)? · 재브리핑 주기(상류 갱신마다 vs 15~30분 정시)? · 발송서버 상시가동 운영 부담 · iOS 웹푸시 제약(홈화면 추가) 수용 여부.

**진짜 선행작업(이것만 풀면 나머지는 재사용):**
1. **#2 TAF/SIGMET 원문 보존** — 상류 KMA API가 원본 TAC를 주는지 확인 + 파서가 버리지 말고 저장. (A묶음 중 유일한 백엔드 작업)
2. **#8 KOCA 라이선스** — AIP 최저치를 도구에 담아도 되는지 서면 확인. 착수 전 게이트.
3. **#4 KIM cutoff** — KMA 명세에서 초기장 시각 확인(작음).

**#7 로그인 — 확정.** 개인 미니마 저장은 이미 localStorage로 되어 있어(#8 설정 포함) **출발점이 있음**. 로그인은 그 위에 (a) 사용자별 프로파일 (b) 다중기기 동기화 (c) #13 푸시 구독 타깃팅 **및 서버측 비행계획 저장** (d) 역할모드를 얹는 열쇠. Express에 인증 미들웨어가 없으니 최소 인증(세션 쿠키 또는 JWT)만 신규로 세우고, 미니마·설정·비행계획 저장을 로그인 후 서버로 승격한다.

**#13은 사실상 #7 위에 서는 별도 서브시스템:** 서버가 계획을 알고(→#7) 주기적으로 재브리핑해 diff → 변화 시 Web Push. 앞선 조사에서 "모바일 푸시=Phase 2"라 했던 건 *공항 임계값 알림* 기준이었고, 이 경로-예보변화 알림에서는 **푸시가 본체**다.
