## 1. 정체성
> 조종사·관제사·기상 담당자가 **비행 전 위험 기상을 즉시 판독**하도록 돕는 지도 기반 관제 대시보드. 깊이(브리핑 전문성) 우선, 넓이(EFB 기능) 비추구.

## 2. 핵심 원칙
1. **판독성 > 미관.** 보기 좋아도 인식이 느리면 실패.
2. **색 = 의미.** 한 의미 = 한 색. 장식 색·그라데이션 금지, 색은 아껴 쓴다(정상=무채색, 위험만 색).
3. **데이터가 주인공.** 베이스맵은 차분히 뒤로, 데이터 레이어가 떠 보이게.
4. **빽빽하되 위계 있게.** 밀도는 높이되 크기·굵기·간격으로 정리.
5. **상태 먼저, 컨트롤 나중.** 화면은 "무엇을/얼마나 안전한가"부터 답한다.
6. **하나의 시스템, 반응형 표현.** 데스크톱·모바일이 색·글씨·의미는 동일, 배치·터치만 다름.

## 3. 하지 말 것
- 보라/파랑 그라데이션, 장식용 그라데이션
- 의미 없는 강조색, 한 색에 두 의미
- rainbow/jet 컬러맵 (→ viridis/cividis/cmocean)
- 색만으로 등급 구분 (→ 색 + 명도 + 범례 + 해칭 병기)
- 데스크톱 레이아웃을 모바일에 그대로 축소
- 눈에 보일 만큼 진한 그림자, CDN 폰트 의존

## 6. 반응형·레이아웃 (구 ui-responsive-guidelines 흡수)

ProjectAMO는 마케팅 사이트가 아니라 운영 도구다. UI는 조종사·관제사·기상 담당자가 **시간 압박 속에서 중요한 상태를 빠르게 인식하고 자신 있게 행동**하도록 도와야 한다.

**제품 렌즈 — 사용자는 이런 질문을 한다:** 지금 어떤 공항·항로·기상 상황인가? · 현재 상태가 안전/제한/곧 변화인가? · 다음에 중요한 경보·예보 변화는? · 다음에 열 컨트롤·상세는?

**정보 경합 시 우선순위:**
1. 정체성·범위 (공항, 항로, 모드, 활성 컨텍스트)
2. 현재 운영 상태 (flight category, 경보, 제한, 안전 영향 조건)
3. 다음 중요한 변화 (TAF 전환, 경보 발생, 예보 변화)
4. 의사결정 보조 상세 (지도 맥락, 보조 카드, 설정)
→ 저우선 컨트롤·장식이 현재 상태·다음 변화 정보를 가리면 안 된다.

**문제로 간주(CSS 오버플로 없어도):** 중요 상태가 덜 중요한 컨트롤 아래 묻힘 · 핵심 기상/경보 찾기 전 카드 너무 많이 스캔 · 지도와 데이터가 주 작업 없이 경합 · 모바일이 데스크톱 구조 유지로 읽기 어려움 · 표/카드가 들어가긴 해도 너무 빽빽 · 핵심 정보 도달에 인터랙션 과다. (전통적 실패: 클리핑·오버랩·잘림·숨은 컨트롤·깨진 클릭·페이지 가로스크롤도 포함.)

**반응형 철학:** 모든 데스크톱 표면을 태블릿/모바일에 한꺼번에 유지할 필요 없다. 좁은 화면에선 — 작업 분리 > 빽빽한 공존 · 모드 전환 > 읽기 힘든 혼합 · 전용 읽기 뷰 > 데스크톱 나란히 구성 유지. 좁아져서 읽기 어려우면 폰트 축소·간격 압축보다 **구조 단순화**.

### 모바일 8원칙 (정식 목표)
1. **한 화면 한 작업.** 데스크톱 패널/모달/드로어를 줄여 끼우지 않는다. (monitoring 화면이 참조 모델)
2. **상태 먼저, 컨트롤 나중.** 모든 주요 화면은 정체성·현재 상태부터. 빈 지도는 상태 화면이 아니다.
3. **상단 작업 탭이 공통 골격.** 슬라이드 아이콘 레일 대신 일관된 상단 작업 전환기.
4. **지도는 작업, 배경이 아니다.** 모바일에서 라이브 지도 위에 데이터 패널/폼/모달을 고정하지 않는다.
5. **공항 상세는 풀스크린, 모든 섹션 1탭.** 상단 탭바에 전 섹션 라벨(현재날씨/METAR/TAF/AMOS/공항경보/기상정보) 동시 노출. 현재날씨는 flight category+경보부터, 공항경보엔 카운트 배지.
6. **빽빽한 데스크톱 콘텐츠는 줄이지 말고 재구성.** TAF·AMOS·미니마는 세로 스택 읽기 블록으로. 가로 스크롤은 폴백, 폰트 축소 금지.
7. **읽기 표면은 한 줄 요약부터.** 서술형 해설·체인지로그는 요약/심각도 한 줄 먼저.
8. **장식은 상태에 종속.** 마스코트·일러스트는 flight category·경보·정체성보다 시각 비중이 높으면 안 된다.

### CSS·레이아웃 규칙
- 폭·간격·breakpoint 추가/변경 전 **공유 토큰(`layoutTokens.css` + 본 헌법 §5)부터 확인.** 새 고정 픽셀 폭은 정당화 없이 추가 금지.
- CSS로 구조 문제를 가리지 말 것. **나쁜 수정:** 빽빽한 데이터가 들어맞을 때까지 폰트 축소 / 충돌 안 나게 간격만 축소 / 데스크톱 표를 좁은 모바일에 그대로 / 읽기 어려운 지도+패널 나란히 유지. **좋은 수정:** 정보 위계 변경 / 작업별 모드 분리 / 보조 컨트롤 뒤로 / 데스크톱 읽기 표면을 모바일 읽기 흐름으로 전환.

### 스크롤 규칙
- 페이지 가로 스크롤 = 실패.
- 패널 내부 가로 스크롤은 본질적으로 2차원인 콘텐츠(빽빽한 표, 다일 예보 그리드)의 폴백으로만 허용. 더 명확한 구조가 있으면 그쪽 제안.

## 7. 워크플로우 (구 ui-responsive-guidelines 흡수)

**보이는 UI 작업 절차:** (1) 관련 상태·뷰포트 캡처 → (2) 편집 전 이슈 기록 → (3) 기계적 깨짐 vs 운영 명료성 문제 분리 → (4) read-only 리뷰어로 점검 → (5) 이슈 수집 후에만 집중 수정 → (6) 재캡처·상태 갱신.
반응형 작업은 영향받는 패널/탭 상태마다 **스크린샷 증거** 필수. `artifacts/responsive-screenshots/<phase>/<YYYY-MM-DD_HHMM_label>/`에 저장 + 짧은 manifest + `review/issues.md`.

**리뷰어 질문:**
- *UI QA:* 핵심 상태를 몇 초 안에 찾나? · 압박 속 읽기 어려운 값/라벨/경보? · 컨트롤이 보이고 닿고 명확한가? · 클릭 가로채기·숨은 상태·오버플로? · 지도가 주 읽기를 압도하지 않나?
- *디자인:* 밀도가 운영 도구다운가? · 타이포가 스캔을 돕나? · 색·강조·그룹핑이 기능적인가? · 간격 폴리시 문제인가 정보구조 문제인가? · 더 강한 구조 변경이 속도·이해를 높이나?
- *스펙:* 승인 범위 안인가? · 미승인 구조 행동을 넣었나? · 제안이 승인 전 구현됐나?

### Proposal-First 규칙 (구조 변경)
새 모바일 작업탭 모델, map/data 모드 분리, drawer→sheet 전환, 풀스크린 스텝 흐름 등 **인터랙션 아키텍처 변경은 기본 미실행.** 캡처·분석·결함 식별·운영자 중심 제안서·기대 효과 정리까지만 하고, **사용자 명시 승인 후** 구현한다.

## 8. 적용 로드맵
- **준비:** 헌법 확정(=이 문서) · Pretendard 자체 호스팅 · 토큰 단일 소스 구축 · lint 가드레일 · 기준선 캡처.
- **계획:** 파일럿 1개 기능 전환으로 패턴 확립 · 컴포넌트 매핑(Fluent vs bespoke) · 마이그레이션 순서(브리핑=견본 → 공항패널 → 날씨오버레이 → 지도 chrome → 모니터링).
- **실행:** 기능 단위 루프(캡처 → 토큰/Fluent 교체 → 스크린샷 회귀 → AA → build/test → 머지). spec→구현→리뷰.

## 9. 작업 후 유지
- 파일 이동·역할 변경·새 규칙·작업 흐름 변경 시 이 문서와 `Architecture.md` 갱신.
- 드리프트(개별 화면이 토큰과 어긋남) 발견 시 → 개별 수정 말고 **토큰에 반영**.
- 줄을 더하기 전에 뺄 줄이 있는지 확인. 빠르게 스캔되게 유지.
