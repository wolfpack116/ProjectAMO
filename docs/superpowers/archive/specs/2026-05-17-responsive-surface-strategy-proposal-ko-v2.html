기존 캡처 이미지를 함께 보여주면서 어느 지점이 문제인지와 왜 개선이 필요한지 설명하기 위한 검토용 리포트입니다. 구현 계획이 아니며, 이 문서만으로 CSS/JSX 변경을 시작하지 않습니다.

주의: 이 문서는 현재 상태를 진단하는 검토 리포트입니다. 결정 사항은 '보류 항목' 섹션을 참고하세요.
★

핵심 요약

디바이스 유형별 전략 방향

모바일 폰

Task-based 화면

작은 화면에서는 지도, 상세정보, 설정을 억지로 같이 넣기보다 한 번에 하나의 주요 작업을 크게 보여주는 방향을 강한 후보로 둡니다.

태블릿

별도 Surface 검토

폰처럼 무조건 분리하지 않고, 지도와 핵심 정보의 동시 표시를 중요한 후보로 유지합니다.

데스크탑 웹

Operations Dashboard

넓은 화면에서는 다중 패널과 높은 정보 밀도를 허용하되, 빠른 스캔 가능성은 계속 검토합니다.

아직 어느 화면을 1급 타깃으로 확정하지 않습니다. 현재 결정할 것은 "폰, 태블릿, 데스크탑을 분리해서 검토한다"는 정도입니다.
1

Monitoring: 폰에서는 정보와 지도가 둘 다 눌림

P1 즉시 검토 필요

P1
지도 렌더링 미작동

폰 크기에서 오른쪽 지도 column이 완전히 검은 배경만 표시됩니다. "좁은 지도"가 아니라 지도 타일 자체가 렌더링되지 않는 수준입니다. CSS 레이아웃 조정만으로 해결되지 않을 수 있습니다.

P1
하단 콘텐츠 접근성 위험

리뷰어는 dashboard root가 viewport보다 길어질 수 있는데 document 높이가 viewport에 묶여 있어 하단 콘텐츠 접근성이 위험하다고 지적했습니다.

개선 방향

폰

기상정보 / 지도 / 설정처럼 task 분리를 강한 후보로 둡니다. 기본 진입은 기상정보가 적절합니다.

태블릿

바로 폰 모델을 복사하지 않습니다. 지도와 정보가 함께 보이는 더 나은 tablet layout을 먼저 비교합니다.

데스크탑

dashboard-style을 유지하되 density와 scan speed를 계속 검토합니다.

2

태블릿 Monitoring: 공존 가능성은 남겨두되 검토 필요

P2 추가 evidence 수집 후 결정

검토 방향

태블릿

폰처럼 무조건 분리하지 않습니다. 특히 1180x820 가로 화면에서는 지도와 핵심 기상정보가 함께 살아날 수 있는지 먼저 확인해야 합니다.

다음 배치

1180x820 태블릿 가로와 820x1180 태블릿 세로를 같은 중요도로 비교하는 것이 좋습니다.

3

메인 지도: 폰에서 패널이 열리면 지도는 거의 배경이 됨

P1 즉시 검토 필요

개선 방향

폰

지도 모드와 상세/설정 모드를 분리하는 후보를 둡니다. 단, Route Briefing 화면에서 우측 상단에 미니맵 썸네일이 이미 존재하는데 — 이 패턴은 살릴 가치가 있습니다. 완전 분리보다 썸네일을 확장한 split 레이아웃이 나을 수 있습니다.

태블릿

지도와 패널 공존 후보를 유지합니다. 다만 패널이 실제 지도 사용성을 해치면 compact panel, overlay, mode split을 비교합니다.

데스크탑

현재 multi-panel map workflow를 유지하되 map occlusion을 계속 확인합니다.

4

Airport Panel: 기계적으로는 개선됐지만 첫 판단 흐름은 더 좋아질 수 있음

P2 폰 흐름 검토 후 결정

개선 방향

폰

공항 사진 헤더가 실제 스크린의 약 45%를 차지해 METAR 데이터까지 도달하기 전에 스크롤이 필요합니다. 헤더를 collapsible하거나 모바일에서 높이를 40~50px로 제한하는 것이 필요합니다. 또한 세로 side tab (METAR/TAF/AMOS)이 터치 타깃이 너무 좁습니다 — 수평 top tabs으로 교체를 강한 후보로 둡니다.

폰 (flow)

full-screen step flow도 후보로 유지합니다. 추천 순서: 요약 → METAR → TAF → Warnings → supporting info

태블릿

full-screen flow가 자동 정답은 아닙니다. 지도 맥락을 유지할 수 있는 넓은 drawer 또는 panel 구조도 후보로 둡니다.

제외

현재 검토에서는 airport bottom tab과 segmented-control 대안은 우선 후보로 보지 않습니다.
