# 모바일 UI/UX 정비 — 설계 스펙

- 작성일: 2026-07-01
- 대상 표면: 지도/레이어 · 공항 상세 · 브리핑 (monitoring·더보기 제외)
- 게이트: 디자인 헌법 §7 Proposal-First (구조 변경은 별도 명시 승인 후 구현)
- 최종 검수: `frontend-design-audit` 스킬

---

## 1. 배경 / 목적

데스크톱은 디자인 헌법(`docs/design/design-language.md`)에 맞춰 정비 완료. 모바일은 미정비.
모바일 3표면을 캡처→분석→제안→승인→구현→audit 흐름으로 헌법에 맞춘다.

이 스펙은 두 덩어리를 포함한다:
1. **헌법 갱신** — §6 모바일 8원칙을 AI가 판정 가능한 6대원칙으로 교체 + §5 토큰 추가.
2. **모바일 audit 워크플로우** — 증거 기반, 병렬 분석, 2버킷 제안.

## 2. 성공 기준 (일관성 3축)

1. **헌법 준수** — §5 토큰 + §6 대원칙(아래 4절)
2. **데스크톱 패리티** — 같은 의미 요소의 색·타이포·라벨·컴포넌트가 데스크톱과 일치
3. **세 화면 상호 일관** — 헤더·시트·칩·1차버튼·탭 패턴이 3표면 간 통일

최종 수용 게이트: `frontend-design-audit` 통과.

## 3. Fluent 적용 범위 (locked 확인)

- **토큰(간격·타이포·radius·중립/시맨틱색): 모바일에도 일괄 적용.** 모바일이 토큰을 어긴 부분이 결함.
- **Fluent React 컴포넌트(`@fluentui/react-components`): 일괄 스왑 불가.** 모바일/터치 전용 세트가 아님 → 헌법 §0 하이브리드대로 일반 UI만 선택 채택, 지도/기상/시트는 자체 컴포넌트 + 동일 토큰.
- 모바일 컴포넌트 프레임워크(Konsta UI·Ionic 등) **도입 금지** — Fluent(§0 locked) 교체 + 불필요 의존. 패턴 참조만.

## 4. 헌법 §6 교체안 — 모바일 6대원칙 (AI 검증용)

> 각 원칙은 `{스크린샷 + 소스}`로 위반/준수를 판정할 수 있게 임계값을 명시한다. 임계값은 §5 토큰에 바인딩한다.
> 근거: Nielsen 10 휴리스틱 + iOS HIG / Material + 터치·WCAG 실무 표준.

### P1. 플랫폼 관습을 따른다
**원칙:** OS 네이티브 패턴(하단 탭·바텀시트·스와이프·풀스크린 상세)을 우선. 커스텀 인터랙션은 네이티브로 표현 불가할 때만, 정당화와 함께.
**AI 검증 체크:**
- 내비게이션이 하단 탭바 또는 표준 시트/드로어 패턴인가? 커스텀 슬라이드 레일·비표준 제스처면 **위반**
- 라이브 지도 위에 데이터 폼/모달을 고정했는가? 바텀시트/풀스크린으로 분리 안 됐으면 **위반**
- 닫기/뒤로가기가 표준 위치(좌상단 또는 시트 핸들)인가? 아니면 **주의**
**검증법:** 소스 내비 컴포넌트 구조 + 스크린샷의 지도 위 오버레이 고정 여부.

### P2. 상태 가시성 우선
**원칙:** 화면은 컨트롤·장식보다 현재 운영 상태(flight category·경보)부터 답한다.
**AI 검증 체크:**
- 첫 뷰포트(스크롤 0, 390×844)에 flight category 또는 경보 상태가 보이는가? 안 보이면 **위반**
- 상태 요소가 상단 1/3 + 크기 ≥ `--fs-400`(16px)인가? 컨트롤/장식이 더 큰 비중이면 **위반**
- 빈 지도/빈 리스트만 뜨고 상태 요약 한 줄이 없는가? 있으면 **위반**
**검증법:** 스크린샷 상단 1/3 스캔 + 소스 JSX 렌더 순서.

### P3. 점진적 노출 & 미니멀
**원칙:** 한 화면 한 작업. 핵심만 먼저, 상세는 펼친다. 데스크톱 밀도를 줄여 끼우지 않고 재구성한다.
**AI 검증 체크:**
- 한 화면에 독립 작업이 2개 이상 동시 노출(예: 지도+폼+리스트)? **위반**
- 읽기 표면(TAF·AMOS·해설)이 한 줄 요약/심각도부터 시작하는가? 원문 덤프 먼저면 **위반**
- 데스크톱 표를 그대로 축소해 폰트 < `--fs-200`(12px)로 넣었는가? **위반**(재구성 필요)
- 페이지 가로 스크롤 존재? **위반**(패널 내부 표 폴백만 허용)
**검증법:** 스크린샷 스캔 + 소스 폰트 크기·`overflow-x`.

### P4. 일관성 & 표준
**원칙:** 같은 의미 요소는 화면 간·데스크톱과 같은 토큰·모양·위치·동작.
**AI 검증 체크:**
- 색·간격·radius·타이포가 전부 §5 토큰에서 오는가? 하드코딩 px/hex 있으면 **위반**
- 3표면의 헤더·시트·칩·1차버튼이 동일 컴포넌트/패턴인가? 표면마다 다르면 **위반**
- 같은 의미(경보=red, VFR=green)에 §5 시맨틱색만 쓰는가? 임의색이면 **위반**
- 데스크톱 대응 요소와 색·타이포·라벨이 일치하는가(패리티)? 어긋나면 **위반**
**검증법:** 소스 grep으로 하드코딩 hex/px + 3표면 CSS 클래스·컴포넌트 대조.

### P5. 터치 인체공학
**원칙:** 주요 액션은 엄지 도달 범위, 오작동 없는 크기·간격.
**AI 검증 체크:**
- 모든 탭 가능 요소의 히트 영역 ≥ `--touch-min`(44×44px, Apple HIG 최소)? 미만 **위반**
- 인접 터치타깃 간격 ≥ `--space-s`(8px)? 미만 **위반**
- 주 액션(탭바·1차 CTA)이 화면 하단 60% 엄지존인가? 상단 코너 단독이면 **주의**
**검증법:** 소스 `min-height/width·padding` 계산 + 스크린샷 간격 측정 + **axe(target-size 룰) 자동 스캔**.

### P6. 접근성은 기본값
**원칙:** 대비·동적 타입·스크린리더·색 이외 단서가 기본값.
**AI 검증 체크:**
- 텍스트/배경 대비 ≥ WCAG AA(본문 4.5:1, 큰 텍스트 3:1)? 미만 **위반**
- 등급/상태가 색만으로 구분되는가? 색+아이콘/라벨/명도 병기 아니면 **위반**(헌법 §3 일치)
- 아이콘 단독 버튼에 접근성 이름(aria-label/텍스트)이 있는가? 없으면 **위반**
- 폰트/간격이 고정 px라 확대에 안 따르는가? 상대단위 권장, 고정이면 **주의**
**검증법:** **axe(color-contrast·aria·label 룰) 자동 스캔** + 스크린샷 대비 확인 + 소스 `aria-*`·단위.

### 6-B. 적용 예 (참고, 규칙 아님)
기존 8원칙의 구체 지침은 대원칙의 예시로 강등한다:
공항 상세=풀스크린+전 섹션 1탭(P1·P3) · 하단 작업 탭바(P1) · TAF/AMOS 세로 스택 읽기 블록(P3) · 읽기 표면 한 줄 요약 먼저(P3) · 마스코트/장식은 상태보다 낮은 비중(P2).

## 6. 도구 추가
- **`@axe-core/playwright`** (devDependency 1개). 캡처 스크립트에 접근성 스캔을 붙여 P5(target-size)·P6(color-contrast·aria) 위반을 객관 수치로 산출.
- Lighthouse·stylelint·jsx-a11y는 이번 범위 밖(과함).
- audit 실행 체계는 이미 설치된 스킬 사용: `design:accessibility-review`, `design:design-critique`, `frontend-design-audit`.

## 7. Audit 워크플로우 (접근안 A: 증거 우선 + 병렬 분석)

### 7.1 캡처
- **뷰포트:** 모바일 390×844(주) + 데스크톱 1536×864(패리티 참조). 폭 종속 결함 발견 시에만 360/430 추가.
- **상태(표면별):**
  - 지도/레이어: 베이스 · 항공시트 · 기상시트 · AIRMET리스트 · 공항경보리스트 · 타임라인활성
  - 공항 상세: 6개 탭(현재날씨/METAR/TAF/AMOS/공항경보/기상정보)
  - 브리핑: 경로입력 · 위험요약결과 · VFR자동생성 · 수직프로파일
- **방법:** `frontend/scripts/`에 모바일 전용 캡처 스크립트 신설(기존 `responsive-screenshots.mjs`는 baseline 3라우트만 커버, 패널/탭/시트 미개방). 각 상태에서 axe 스캔 동시 실행 → JSON 저장.
- **절차:** `docs/dev-server-and-capture.md` 준수(포트 5173 `--strictPort`, DOM ready selector, `networkidle` 금지).
- **저장:** `artifacts/responsive-screenshots/mobile-audit-2026-07-01/<state>/` + manifest(커밋·뷰포트·방법·axe 요약).

### 7.2 병렬 분석
- 표면당 read-only 리뷰어 서브에이전트 3개.
- 각 입력: `{모바일샷 + 데스크톱샷 + axe JSON + 소유 소스 목록 + 헌법 §5/§6(6대원칙) + 3축 체크리스트}`.
- 각 출력: `review/issues-<surface>.md`. 결함 1건 = `{상태, 스크린샷, 위반 원칙(P1~P6)/축(헌법/패리티/상호), 심각도(기계적깨짐/운영명료성/폴리시), 증거, 수정방향, 분류힌트(토큰/구조)}`.
- 회수: 요약 600자 이내(롱컨텍스트 §5.5). 본문은 파일로.
- **상호 일관성은 메인이 3보고 취합 후 4th 패스로 판정**(서브는 서로 못 봄): 헤더·시트·칩·버튼·탭 패턴 대조.

### 7.3 제안서 (승인 게이트)
- `docs/superpowers/specs/2026-07-01-mobile-design-audit-proposal.md`.
- 결함을 2버킷 분류:
  - **버킷1 토큰/시각:** 간격·색·타이포·터치타깃·그림자·기존 구조 내 위계 → 승인 후 바로 구현
  - **버킷2 구조:** 섹션 재배치·시트↔풀스크린·탭 모델·drawer→sheet 등 인터랙션 아키텍처 → **§7 별도 명시 승인**
- 각 항목: id · 표면 · 증거샷 · 위반 원칙 · 심각도 · 수정방향 · 버킷 · 예상 공수.
- **여기서 멈추고 사용자 승인 대기.**

### 7.4 구현 규칙
- 토큰 드리프트는 개별 화면 아닌 **토큰/공유 CSS에 반영**(헌법 §9).
- 맵 레이어/오버레이/타임라인 로직은 **소유 feature 모듈 `useXOverlay` 훅**(ADR 0001) — MapView.jsx에 state/useEffect 추가 금지.
- **인코딩 안전**: `apply_patch` / `fs.writeFileSync(utf8)`. PowerShell Set-Content/Out-File/`>` 금지(헌법 §6).
- 버킷1 먼저, 버킷2는 승인 후.
- 코드 수정 후 `graphify update .`.

### 7.5 검증 & 핸드오프
- 재캡처(after) + axe 재스캔 → before/after 대조.
- **`frontend-design-audit` 최종 검수.**
- 롱컨텍스트 §3 status 파일(`docs/superpowers/status/mobile-design-audit.status.md`) 유지, 세션 종료 전 Resume Point 갱신.
- 종료 시 `Architecture.md` File Roles 갱신(신설 캡처 스크립트·헌법 변경 반영).
