Airport Panel과 Route Briefing의 개선안 와이어프레임은 docs/superpowers/specs/2026-05-17-responsive-surface-strategy-proposal-ko-v2.html의 개선안 시각화 섹션을 주 기준으로 사용합니다. 이 파일의 단순 와이어프레임은 보조 설명용입니다.

Surface별 기본 방향

모바일 폰

후보한 번에 하나의 주요 작업을 크게 보여주는 task-based 구조.

태블릿

후보지도와 핵심 정보의 동시 표시를 우선 검토. 폰 구조를 바로 복사하지 않음.

데스크탑

유지operations dashboard 성격 유지. 다중 패널과 높은 정보 밀도 허용.

1. Monitoring: 폰에서 정보와 지도 모두 좁아지는 문제

P1지도는 실제 환경에서 보인다고 가정합니다. 문제는 지도와 기상정보를 같은 좁은 화면에 같이 넣으면서 둘 다 약해지는 것입니다.

구현 후보

  • 폰에서는 기상정보 / 지도 / 설정 task tab 구조를 prototype으로 구현합니다.
  • 태블릿에는 이 구조를 바로 적용하지 않습니다. 별도 캡처 후 결정합니다.

2. 태블릿: 지도+정보 공존을 먼저 검토

다음 캡처에서 봐야 할 것

  • 1180x820 태블릿 가로에서 지도와 기상정보가 같이 읽히는지
  • 820x1180 태블릿 세로에서 지도 영역이 너무 좁지 않은지
  • mode, settings, exit control이 빠르게 발견되는지

3. Airport Panel: compact header + horizontal top tabs 후보

P2이전에는 full-screen step flow만 강하게 보았지만, 시각 검토 결과 폰에서 compact header와 horizontal top tabs도 좋은 후보로 유지합니다.

4. Route Briefing: split layout 실험 후보

실험완전 분리보다 form과 live map context를 함께 보여주는 split layout을 작은 prototype으로 검토합니다. 맞지 않으면 폐기 가능한 범위로 구현합니다.
